# マップ・拠点・ステージ

拠点／占領、エネミー＆ステージ構成といった「マップ側」の設計メモ。

> 関連: 戦闘解決は [combat.md](combat.md)／ユニット性能・対応表は [units.md](units.md)。

## 勝敗条件

- 原典（ネクタリス）は **「敵全滅」または「敵拠点の占領」を50ターン以内に達成** で勝利。
- **本作の暫定実装【暫定】**:
  - **自軍全滅＝敗北**。相討ちで両軍が同時に全滅した場合も、自軍が消えていれば **敗北を優先**（原典は相討ち全滅の結末を明示しておらず、拠点占領＋ターン制限でこの事態を実質回避している。本作は未実装ぶんを素直に「負け」で塞ぐ）。
  - **敵全滅＝勝利**。
  - **拠点占領による勝利・ターン制限は未実装**（拠点占領を入れれば相討ち全滅はレアケース化する）。【未決】

## 拠点・占領

- **拠点＝城・砦（軍事拠点）** と定義する。【確定】
- **占領ユニットを3層化** し、占領可能な高ステータス枠で凝ったステージを作る。【確定】
  - **クレリック**：安価で脆い（数で押す／使い捨て寄り）。
  - **ビショップ**：高耐久で保持向き（占領した拠点を守る）。
  - **パラディン**：戦えるエリート（占領しつつ前線でも機能）。
- **占領＝駐留ユニットの寝返り（捕虜）**：拠点には敵ユニットが駐留しており、占領するとそのユニットは **同種の味方ユニットに変わって自軍に加わる**（例: ゴブリン → クレリック）。世界観上は「拠点に囚われていた**捕虜**が解放されて味方につく」イメージ。
  - これが **ドラゴンを占領で奪えない理由**にもなる：ドラゴンは敵専用で **対応する味方ユニットが存在しない** ため、寝返らせる先がない＝占領しても自軍に加わらない。
- **残る論点【未決】**：寝返るユニットの兵数（満員で寝返るか／元の残兵を引き継ぐか）、空の拠点を占領した場合の扱い、寝返り数の上限など。

## ステージデータの構成

- ステージは JSON データで記述する（ASCII地形グリッド＋ユニット配置）。ローダーは `data/stage_loader.gd`（`StageLoader`）。詳細 → [../tech/architecture.md](../tech/architecture.md)。
- **冒険譚（キャンペーン）ごとにフォルダ**で束ねる: `data/stages/<冒険譚>/<ステージ>.json`。
  - 1つの冒険譚＝数ステージ構成を想定し、その単位でフォルダに放り込む。
  - 例: 動作確認用のデモは `data/stages/demo/demo.json`。
- 【未決】冒険譚内のステージ順序・進行管理（マニフェスト等）、ステージ選択。当面は読込先を決め打ちで1枚。

## エネミー／ステージ設計

- **テーマ別リスキン方式**：フィールドのテーマ（古代遺跡・死霊の洞窟…）ごとに、**同ステータス・別名のエネミーセット**を用意する。【方針確定】
  - 標準ロスター ＝ **テーマ0** と位置づける（→ [units.md](units.md) の対応表）。
  - マップに実際に出る分だけ命名すればよい（全テーマ×全種を先に作り込む必要はない）。
