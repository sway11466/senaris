# マップ・拠点・ステージ

拠点／占領、エネミー＆ステージ構成といった「マップ側」の設計メモ。

---

## 用語（冒険譚・ステージ・シナリオ）

物語まわりで混同しやすい3語を定義する。**冒険譚とステージは「入れ物」、シナリオはそこに乗る「中身（物語）」** という役割分担。【確定】

| 用語 | 意味 | 粒度 | データ上の実体 |
|---|---|---|---|
| **冒険譚**（campaign） | 物語で束ねた一連のステージ群。配布・選択の単位 | 大 | `data/stages/<冒険譚>/` ＋進行マニフェスト |
| **ステージ**（stage） | 1回の戦闘＝1マップ。読み込んで遊ぶ最小単位 | 中 | `<ステージ>.json`（盤面＋ユニット配置＋勝敗条件＋戦力供給モデル） |
| **シナリオ**（物語・演出） | 冒険譚やステージに乗る筋書き・ブリーフィング・会話。コンテナではなく**側面語**で、固有の粒度を持たない | — | 各粒度に付随（「この冒険譚のシナリオ」「ステージ3のシナリオ」のように使う） |

- 例: チュートリアルの「町がゴブリンに襲われる」物語＝**1冒険譚**、その各マップ＝**ステージ**、各マップの筋＝**シナリオ**。

## 勝敗条件

- 原典（ネクタリス）は **「敵全滅」または「敵拠点の占領」を50ターン以内に達成** で勝利。
- **本作＝ステージごとに勝敗条件を「条件リスト」で設定し、いずれか1つで決着（OR）**。【方針確定】
  - **勝利条件**（ステージが組み合わせる）: **殲滅**（敵が盤上から消え、かつ復帰手段＝出せる garrison も無い）／**敵本拠地の占領**（`hq` の敵拠点を取る）／**ボス撃破**（指定ユニット）…（特定hex確保などは追って）。
  - **敗北条件**: 自軍が盤上0 かつ 復帰手段なし（自軍所有の拠点に、出撃できる garrison が1体も無い）／**味方本拠地を奪われる**／**ターン制限を超過**（時間切れ＝引き分けなしで敗北）。勝利の殲滅と対称。拠点の中の控えはまだ生きている戦力として数える＝盤上が空でも、出せる駒がある限り負けにはしない。
  - **殲滅は「盤上＋復帰手段」で数える**：拠点の中の眠り駒（garrison）は盤上に居ないが、出せる控えがある限りその陣営は消滅していない＝決着しない。盤上だけを空にしても敵保持の拠点から出撃してくるので、勝ちきる（敵を消滅させる）には実質その拠点の制圧が要る。
  - **相討ち全滅**は自軍が盤上から消えていれば敗北優先（暫定踏襲）。
  - この「盤上＋復帰手段」で勝敗を見る方式を案Bと呼ぶ。副作用として、盤上0でも控えを出さず引き籠ると膠着しうるため、ターン制限で決着させる（＝時間切れ敗北。実装済み・下記）。
- **実装状況**: 敵全滅／自軍全滅は実装済み。**条件リスト化・ボス撃破・本拠地占領も実装済み**＝ステージJSONの `victory` 配列（OR評価）を `BattleState.victory_conditions` が判定。
  - ボス撃破 `{ "type": "defeat_unit", "unit_id": N }`（ボスは駒に `"id": N` を明示採番。enemy 部隊の `units[]` 内でよい）。デバッグステージ: `data/stages/debug-victory/boss.json`。
  - 本拠地占領 `{ "type": "capture_hq" }`＝敵 native の `kind:"hq"` 拠点をすべて自軍が保持で勝利（`Base.kind`・`native_team` を追加。該当 hq が無ければ不成立＝空勝ち防止）。**自軍 native の hq を奪われたら敗北**（常時ルール・hq を置いたステージだけ効く。奪還で解消）。デバッグステージ: `data/stages/debug-victory/hq.json`。占領は移動の瞬間に起きるため、決着チェックは移動直後にも走る（MatchController）。
  - 殲滅勝ち/全滅負けは条件リストと無関係に常時有効（相討ち・hq喪失は敗北優先）。**閉じ込め判定（案B）は実装済み**＝消滅を「盤上0 かつ 復帰手段なし」で判定（`_has_reinforcement`）。
  - **ターン制限も実装済み**＝ステージJSONの `turn_limit`（1手番＝両陣営1巡）。`turn_number > turn_limit` で**時間切れ敗北**（引き分けなし・勝利条件を満たしていればそちらが優先）。**実ステージJSONでは必須**（`load_file` が未指定/0以下を push_error＝データのバグ扱い）。既存ステージは一律 30。`build` は未指定を素通し（0＝無制限。合成テスト用）。決着チェックは `MatchController.end_turn` でも走る（ターン跨ぎで発火）。

## 拠点・占領

- **拠点＝城・砦（軍事拠点）** と定義する。
- 拠点は4つの機能を持つ: **①防御地形 ②占領（即時） ③出撃（ネクタリス方式） ④回復（休憩）**。

### 拠点＝地形（防御に強い）

- 拠点は **防御の強い地形タイプ**（`fort`＝拠点）として `terrain.csv` に持つ。攻×1.0／**防×1.2**（守り特化・要チューニング）。移動は地上コスト1（普通に入れて占領できる）。
- 見た目・地形補正・移動コストは既存の地形システムに乗る（[combat.md](combat.md) 地形効果／[movement.md](movement.md)）。占領・出撃・回復のロジックは地形とは別レイヤー（後述）で持つ。
- **「占領可能な拠点かどうか」は地形ではなく `Base` の有無で判定する**（`state.base_at(hex)`）。地形 `fort`/`castle` は防御＋見た目だけで、`Base` とは**自動連動させずステージで重ねて配置する**【現状維持の方針】＝`Base` の無い fort は単なる防御地形、平地の上の `Base` も占領可。実装は [base.gd](../../domain/capture/base.gd)・`BattleState`。

### 拠点の種類（kind）・所有者（team）・ユニットの生来（native）

- **`Base.kind`（不変）＝ 味方／敵／中立**、＋ **`hq`（本拠地フラグ）**。これと **現在の所有者 `team`（可変）は別軸**：占領で team は変わるが kind は変わらず、**kind が挙動を決める**。
- **ユニットは生来の陣営 `native_team`（味方／敵／中立。JSONキーは `native`）を持つ**（新規データ）。拠点の中の駒がどう振る舞うか（寝返るか・閉じ込められるか）は、拠点の所有者ではなく**ユニットの native** で決まる。
- 種類ごとの性格:
  - **味方拠点**: garrison は友軍（native=味方）。自軍が出撃できる味方の控え。
  - **敵拠点**: garrison は敵（native=敵）。敵が出撃させる＝**湧き元**。味方が占領すると湧きを止められる（denial）。
  - **中立拠点**: garrison は中立（native=中立）。**占領した側の戦力になる**（味方が取れば味方として解放／敵が取れば敵として湧く）。

### 占領（即時）

- **占領できるのは兵種「占領兵」**（cleric / priest / bishop / paladin ＝ `can_capture` true）。兵種が占領兵なら占領可、という対応で揃える（unit_type.csv の兵種列と `can_capture` を一致させる）。
- そのうち **ステージ設計の軸になる3アーキタイプ**を据える（残りの priest＝高火力 も占領兵として併用できる）:
  - **クレリック**：安価で脆い（数で押す／使い捨て寄り）。
  - **ビショップ**：高耐久で保持向き（占領した拠点を守る）。
  - **パラディン**：戦えるエリート（占領しつつ前線でも機能）。
- **占領＝即時**: `can_capture` ユニットが敵・中立の拠点hexに**進入した瞬間**に、その拠点の所属が自軍へ移る。別コマンドは作らない（入る＝占領）。駐留敵がそのhexに居る間は進入できない（倒してから入る）。

### 出撃（ネクタリス方式・寝返り／解放）

- 拠点は中に **garrison（眠っている控え駒）** を抱える＝**初期配置ぶん＋回復で中に入ってきた駒**。
- **出撃**: 所有者は garrison から1体を選び、**拠点に隣接する空きhexへ1歩で出す**（出した駒はそのターン行動完了・次ターンから通常行動）。出撃できるのは「**拠点の所有者＝その駒の現陣営**」のとき。
- **出撃→直接乗車**: 拠点に隣接する**乗れる味方輸送のマスも出撃先になる**＝garrison の駒をそのまま輸送に乗せられる（盤上には出ない・そのターンは降ろせない）。詳細 → [movement.md](movement.md)（輸送）。
- **占領時の挙動は native で決まる**:
  - **native=中立** の駒 → 占領した側に**寝返る**（中立拠点の戦力は取った側のもの）。
  - **native=味方／敵** の駒 → **寝返らない**。相手に拠点を奪われると**中に閉じ込められ出られない**（再奪還で再び出せる＝「眠っていた敵が復活」）。だが**消えてはいない**。
- **殲滅カウントは盤上のみ**：中の眠り駒は数えない。味方が敵拠点を制圧すれば中の敵は出られず、殲滅勝利を妨げない（[勝敗条件](#勝敗条件)）。
- **garrison の種別・数はステージデータで指定**（例: 囚われた町人を **ノービス×2**）。**ドラゴンは対応する味方が無い**ため解放対象にできない（native の整合）。

### 回復（休憩）

- 駒は**自軍拠点の中に入って回復**（ネクタリス式）＝hex の“上”に乗るのではなく garrison として中で休み、兵数が max まで回復してから出撃で戻す。**兵数のみ回復、経験Lvは据え置き**。
- **敵拠点（kind=敵）では回復しない**。味方・中立拠点で回復できる。
- 回復で中に入った駒も garrison 扱い＝**閉じ込め／寝返りの native ルールが同様に効く**（味方拠点が奪われれば、中で回復中の味方は閉じ込められる＝出られないが味方のまま）。

### 実装状況（回復・native）

- **回復＝「中に入る」モデルは実装済み**: 自軍所有の拠点hexで移動確認メニューの**「入る」**を選ぶと garrison になり盤上から消える（**選択式**＝上に立って fort 地形で守る選択も残る）。手番開始で**満員へ全回復**（経験Lv据え置き）。出るのは出撃（1歩・行動完了）。**奪った敵 native の拠点では回復しない**（出撃拠点にはなる）。収容数は無制限。**盤上最後の1体でも、入った直後に復帰手段が残るなら入れる**（案B＝`can_enter_base_at`。全周封鎖など入れば即敗北になる場合だけ不可）。
- **`Unit.native_team`（生来の陣営）実装済み**: 既定＝初期陣営、ステージJSONの `"native"` で上書き可。garrison の既定は拠点の初期所属（中立拠点の駒＝中立）。**出撃は native ゲート**＝中立 native は取った側に寝返る／味方・敵 native は「所有者＝生来の陣営」のときだけ出撃可＝**奪われた駒は閉じ込め**（奪還で復活）。閉じ込め駒は回復もしない。

### 残る論点【未決】

- garrison から出撃／解放する駒の兵数＝**満員(8)・Lv1で確定**（解放捕虜は満員で出る）。
- 敵AIの拠点出撃（湧き）は設計確定＝engage 再利用で条件成立時に“出せるだけ”出す（[ai.md](ai.md) §7・実装は別途）。湧きの上限は garrison 頭数で自然に有界（敵は「入る」を使わない＝補充なし）。

## ステージデータの構成

- ステージは JSON データで記述する（ASCII地形グリッド＋ユニット配置）。データは `data/`、それを盤面(BattleState)に組み立てるローダー `StageLoader` は `application/`（data＋domain の両方に依存するため）。詳細 → [../tech/architecture.md](../tech/architecture.md)。
- **冒険譚（キャンペーン）ごとにフォルダ**で束ねる: `data/stages/<冒険譚>/<ステージ>.json`。
  - 1つの冒険譚＝数ステージ構成を想定し、その単位でフォルダに放り込む。
  - 例: 動作確認用のデバッグステージは機能別フォルダ `data/stages/debug-*/`（一覧 → [../tech/debug-stages.md](../tech/debug-stages.md)）。
- 冒険譚内のステージ順序・進行管理（マニフェスト）・ステージ選択 → [stage_select.md](stage_select.md)。実装までは読込先を決め打ちで1枚。

### 駒の配置（陣営セクション）

- 駒は**陣営ごとのセクション**に分けて書く。**このゲームは味方＝人間操作／敵＝AI操作が1:1**なので、陣営で構造を分ける（駒ごとの `team` は書かない＝セクションが陣営を決める）。
  - **`player`**: 自軍の駒の配列。人間が操作する（AI/部隊は持たない）。
  - **`enemy`**: 敵の**部隊(squad)の配列**。**敵は必ずどれかの部隊に属する**（バラ配置は無い）。各部隊は AI プリセット `ai` を持つ。ただの1波は名前なしの `{ "ai": "charge", "units": [...] }` でよい。
- 各駒（`player[]` / `enemy[].units[]`）の記法は共通: `type`（性能）か `skin`（見た目・性能は逆引き）＋ `col`/`row`＋個別上書き（`move`/`troops`/`atk`/`def`/`level`/`range`…）＋任意 `id`（ボス指定など）。輸送は `passengers:[...]`。
- **部隊(squad)** ＝ `{ "name"?: 表示名, "ai": プリセットラベル, ...上書き, "units": [...] }`。`ai` は [ai.md](ai.md) のプリセット（`charge`/`guard`/`raid`/`weak`…）。`guard` は `sight`（索敵半径）等を足せる。
- 例:
  ```json
  "player": [
    { "type": "fighter", "col": 2, "row": 5 }
  ],
  "enemy": [
    { "ai": "charge", "units": [ { "skin": "goblin", "col": 9, "row": 5 } ] },
    { "name": "奥の部屋", "ai": "guard", "sight": 3, "units": [ ... ] }
  ]
  ```

### 陣営の表記（拠点の team / native）

- **拠点・native の陣営は可読な文字列**で書く: **`"player"`（自軍）／`"enemy"`（敵）／`"neutral"`（中立）**。内部は int 規約（0/1/-1）で持つが、データ側では数値を書かない（`StageLoader` が文字列→int に変換）。駒の陣営はセクションで決まる（上記）ので `team` は書かない。
- **`bases[].team`**: 拠点の所属。省略時は `neutral`。
- **`native`**: 駒の生来の陣営（[出撃・寝返り](#出撃ネクタリス方式寝返り解放)を決める）。省略時は各文脈の既定（盤上の駒＝自身のセクション陣営、garrison＝拠点の初期所属、搭乗＝輸送と同陣営）。
- 例（拠点＋garrison）:
  ```json
  "bases": [
    { "col": 9, "row": 6, "team": "neutral",
      "garrison": [ { "type": "novice", "count": 2, "native": "player" } ] }
  ]
  ```
  ↑ **中立拠点だが garrison は `native:"player"`** なので、敵に踏まれても寝返らず自軍の解放待ち（囚われの町人）。

### 戦力供給モデル（独立／継承）

ステージがプレイヤー戦力をどう用意するかを、**ステージ（冒険譚）ごとに切り替えられる**ようにする。【方針確定】「与えられた戦力でやりくりする」（[combat.md](combat.md) 生産なし）の"与えられ方"を、シナリオに応じて変える仕組み。

- **独立（fresh）＝既定**: そのステージが戦力を丸ごと配給する。前ステージの結果を引き継がない。チュートリアル各話のように1話完結のステージはこれ。
- **継承（carryover）**: 前ステージの**生存ユニットを持ち越す**（経験Lv・残兵数つき）。ダンジョン攻略・連戦シナリオ向け。ステージ側で増援を足してもよい。
- データ上はステージのフィールドで宣言する想定（例 `roster: "fresh" | "carryover"`）。冒険譚マニフェストと合わせて進行管理する。

**継承時に決める細目【未決】**:
- 何を継ぐか＝**生存ユニットのみ**／経験Lvは保持／**兵数は残ったまま（回復なし）**か、ステージ間で一部回復させるか。
- キーユニットを失って詰む（ソフトロック）対策＝継承ステージに増援を置くか、敗北救済をどう設計するか。
- セーブ（[../tech/gamesystem.md](../tech/gamesystem.md)）との接続（どのステージから再開し、戦力スナップショットをどう持つか）。

## エネミー／ステージ設計

- **テーマ別リスキン方式**：フィールドのテーマ（古代遺跡・死霊の洞窟…）ごとに、**同ステータス・別名のエネミーセット**を用意する。【方針確定】
  - 標準ロスター ＝ **テーマ0** と位置づける（→ [units.md](units.md) の対応表）。
  - マップに実際に出る分だけ命名すればよい（全テーマ×全種を先に作り込む必要はない）。
