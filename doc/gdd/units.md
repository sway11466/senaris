# ユニット

ユニットの性能設計・陣営方針と、具体ステータス（対応表）。
名前・見た目はオリジナル、性能特性はネクタリスを下敷きにする。

> 関連: 戦闘解決は [combat.md](combat.md)／占領・拠点運用は [map.md](map.md)。

---

## 性能設計

- 名前・見た目はオリジナルのファンタジー風。
- 性能的な特性（移動力・攻撃／防御傾向・地形適性・経験値による成長など）はネクタリスを下敷きにする。
- どのネクタリス的特性を、どのファンタジーユニットに割り当てるかは未決（下記対応表）。
- **完全互換は目指さない**：ネクタリスの8カテゴリ23種をすべて引き継ぐ必要はない。役割は取捨選択してよい。機雷（＝ネクタリスのヤマアラシ）は**不採用**。【確定】
- **両刀（対地対空）ユニットの扱い**：対地・対空の両対応ユニットは、火力・機動・耐久のいずれかで明確に妥協させる（万能で器用貧乏にする）。【指針】
- **パラディン**：万能ゆえに強力なので、**厳選マップ限定**で運用する。【確定】

## 陣営方針：基本ミラー＋例外

基本方針は **「基本ミラー＋例外」**。

1. **敵味方はミラーを土台にする**：性能・役割は敵味方で同じ（名前と見た目だけ陣営別）。【確定】
   - → 役割＋ステータスの表は **1セット作れば両陣営に使える**。
2. **見た目だけ陣営別**：同じ役割でも、名前・グラフィックは陣営ごとに別物にする。【確定】
3. **片陣営専用の強ユニットを少数置いてよい（ハンター枠／例：ドラゴン）**：これがミラーに対する「例外」。
4. **専用強ユニット（ドラゴン）は占領で奪えない（暫定 / ※要確定）**：敵専用の強ユニットは占領しても自軍に編入しない。【暫定】
   - 特別感を保てる。
   - 拠点占領による増援との衝突も回避できる（→ [map.md](map.md)）。

### 補足メモ

- **ドラゴン（敵専用）**: ネクタリスのハンター枠＝万能の強ユニット。味方版なし。占領しても奪えない（暫定）。
- **パラディン**: 万能エリートのため**厳選マップ限定**で運用（万能ユニットはバランスを壊しやすい＝ハンターの教訓）。
- **占領の3層**: クレリック（安価・脆い）／ビショップ（高耐久・保持）／パラディン（戦える・エリート）。→ 運用は [map.md](map.md)。
- **両刀（対地対空）ユニットの妥協**: アーチャー＝低火力、スナイパー＝脆い、ニンジャ＝脆い。パラディンのみ弱点が薄い（要・運用制限）。
- **不採用**: ヤマアラシ（機雷）。完全互換は目指さない方針。

## データ構成（実装）

**性能と見た目を分離する**（アーキの「ステータスは原型に1回、名前・スプライトは上書きレイヤー」）。

- **性能 ＝ `UnitType`**（`data/units/unit_type.gd`）＝ステータスのみ。名前も画像も持たない。
  - 種別表 `data/units/unit_type.json`（テーマ非依存の原型ロスター）。`UnitCatalog`（`data/units/unit_catalog.gd`）が `id → UnitType`。
  - 箱だけ先に用意（実装は将来）: `atk_air`・`attack_range`(>1=間接)・`move_type`。現状の戦闘は `atk_ground`＝攻撃・`defense`＝防御のみ使う。
- **見た目＋識別 ＝ `UnitSkin`**（`data/units/unit_skin.gd`）＝名前・説明・画像。1性能に複数ぶら下がる（陣営別・テーマ別の別名）。
  - スキン表 `data/units/unit_skin.json`（性能とは別ファイル＝上書きレイヤー。スキンは“何のスキンか”の配下に置く）。`SkinCatalog`（`data/units/skin_catalog.gd`）が `type_id → {ally:[UnitSkin], enemy:[UnitSkin]}`。テーマが増えたら `data/units/unit_skin/<テーマ>.json` に割ってよい。
- **配置方針**: ユニットの型・データ・ローダーは `data/units/` に同居（機能フォルダ）。型とデータをセットで扱う。
- **正本はCSV（表計算で管理）**: フラット/グリッドな表は CSV が正本（人間が表計算/VSCodeで編集）。CSV正本・生成JSON・変換ツールは**機能フォルダに同居**（`data/units/` にまるっと）。`data/units/convert.gd`（headless）が **CSV → コード用JSON** を生成。
  - **CSVは2行ヘッダ**: 1行目=英語キー（コードが使う）／2行目=日本語ラベル（人間用・変換時は読み飛ばす）／3行目以降=データ。参考用の列（兵種・備考など）を足してもよい（コードは未知キーを無視）。
  - `data/units/unit_type.csv` → `data/units/unit_type.json`（**生成物・手で触らない**）。
  - `data/units/unit_skin.csv`（1行=1別名: type_id, side, name）→ `data/units/unit_skin.json`。画像・説明は当面空で、必要時にCSVへ列追加。
  - `data/movement/movement.csv` → `data/movement/movement.json`（移動タイプ×地形コスト表 → [movement.md](movement.md)）。
  - 表計算向き＝**ユニット性能・エイリアス・移動タイプ**の3表（1行=1レコードのフラット表）。ステージ(json) は手書きのまま。
  - 実行: `godot --headless --script res://data/units/convert.gd`
  - **同性能・別名**（ゴブリン↔守護像）は enemy 配列にスキンを並べるだけ。**どのスキンを使うかは冒険譚側が決める**（ユニットデータは冒険譚/テーマ名を持たない＝責務分離）。引きは `SkinCatalog.skin(catalog, type_id, team, index)`。
  - 画像スロットと未用意時のプレースホルダは [../art/overview.md](../art/overview.md) 参照。
- **ステージは種別を名前参照**: `{ "type": "cleric", "team": 0, "col": 3, "row": 3 }`（`level`/`troops` 等は任意で上書き）。`StageLoader` が `UnitCatalog` 経由で解決し、`Unit.type_id` に保持（描画でスキンを引く／将来の占領＝寝返りに使う）。
- 現状の実装ロスター/スキンは歩兵5種（cleric/priest/monk/bishop/paladin）。残りは順次。
- 【将来】移動タイプ＝地形移動コスト・地形適性（例: 森を低コストで抜ける）はマップの地形テーブルとセットで別途設計。
- 【将来】アーキ本筋の「原本＝スプレッドシート→CSV→.tres」量産パイプライン（[../tech/architecture.md](../tech/architecture.md)）。当面は JSON で回す。
