# ユニット

ユニットの性能設計・陣営方針と、具体ステータス（対応表）。
名前・見た目はオリジナル、性能特性はネクタリスを下敷きにする。

---

## 性能設計

- 名前・見た目はオリジナルのファンタジー風。
- 性能的な特性（移動力・攻撃／防御傾向・地形適性・経験値による成長など）はネクタリスを下敷きにする。役割は取捨選択してよい（機雷＝ヤマアラシは不採用）。
- **敵味方はミラー＋例外**：性能・役割は敵味方で共通にし、名前・グラフィックだけ陣営別にする。例外として、片陣営専用の強ユニットを少数置いてよい（ハンター枠／例：ドラゴン）。専用強ユニットは占領で奪えない（暫定）＝特別感を保ち、拠点占領による増援との衝突も避けられる（→ [map.md](map.md)）。
- **両刀（対地対空）ユニットの扱い**：対地・対空の両対応ユニットは、火力・機動・耐久のいずれかで明確に妥協させる（万能で器用貧乏にする）。【指針】
- **パラディン**：万能ゆえに強力なので、**厳選マップ限定**で運用する。

## ユニット対応表（テーマ0／標準）

ネクタリスの性能を引き継いだユニット表。**敵味方はステータス・役割がミラー**で、名前とグラフィックのみ陣営別。
この表の「ステータス」が**共有テンプレート**で、「味方／敵」はその最初のスキン。
フィールドのテーマ（古代遺跡・死霊の洞窟など）ごとに、**同ステータス・別名のエネミー列を増やす**運用（＝この表はテーマ0／標準。テーマ別リスキン方式は [map.md](map.md) 参照）。

### 補足メモ

- **ドラゴン（敵専用）**: ネクタリスのハンター枠＝万能の強ユニット。味方版なし。占領しても奪えない（暫定）。
- **パラディン**: 万能エリートのため**厳選マップ限定**で運用（万能ユニットはバランスを壊しやすい＝ハンターの教訓）。
- **占領の3層**: クレリック（安価・脆い）／ビショップ（高耐久・保持）／パラディン（戦える・エリート）。→ 運用は [map.md](map.md)。

## データ構成（実装）

**性能と見た目を分離する**（アーキの「ステータスは原型に1回、名前・スプライトは上書きレイヤー」）。観点ごとに以下。

### 1. 性能と見た目の分離

- **性能 ＝ `UnitType`**（`data/units/unit_type.gd`）＝ステータスのみ。名前も画像も持たない。
- **見た目＋識別 ＝ `UnitSkin`**（`data/units/unit_skin.gd`）＝名前・説明・画像。1性能に複数ぶら下がる（陣営別・テーマ別の別名）。
- **同性能・別名**（ゴブリン↔守護像）は enemy 配列にスキンを並べるだけ。**どのスキンを使うかは冒険譚側が決める**（ユニットデータは冒険譚/テーマ名を持たない＝責務分離）。引きは `SkinCatalog.skin(catalog, type_id, team, index)`。

### 2. ファイル・フォルダ配置

- ユニットの型・データ・ローダーは `data/units/` に同居（機能フォルダ）。型とデータをセットで扱う。
- `UnitType`: 種別表 `data/units/unit_type.json`（テーマ非依存の原型ロスター）。`UnitCatalog`（`data/units/unit_catalog.gd`）が `id → UnitType`。
- `UnitSkin`: スキン表 `data/units/unit_skin.json`（性能とは別ファイル＝上書きレイヤー）。`SkinCatalog`（`data/units/skin_catalog.gd`）が `type_id → {ally:[UnitSkin], enemy:[UnitSkin]}`。テーマが増えたら `data/units/unit_skin/<テーマ>.json` に割ってよい。
- 画像スロットと未用意時のプレースホルダは [../art/overview.md](../art/overview.md) 参照。

### 3. CSV正本パイプライン

- **正本はCSV**（表計算で管理）: フラット/グリッドな表は CSV が正本（人間が表計算/VSCodeで編集）。CSV正本・生成JSON・変換ツールは機能フォルダに同居（`data/units/` にまるっと）。
- `data/units/convert.gd`（headless）が **CSV → コード用JSON** を生成。実行: `godot --headless --script res://data/units/convert.gd`
- **CSVは2行ヘッダ**: 1行目=英語キー（コードが使う）／2行目=日本語ラベル（人間用・変換時は読み飛ばす）／3行目以降=データ。参考用の列（兵種・備考など）を足してもよい（コードは未知キーを無視）。
- `data/units/unit_type.csv` → `data/units/unit_type.json`（**生成物・手で触らない**）。
- `data/units/unit_skin.csv`（1行=1別名: type_id, side, name）→ `data/units/unit_skin.json`。画像・説明は当面空で、必要時にCSVへ列追加。
- `data/movement/movement.csv` → `data/movement/movement.json`（移動タイプ×地形コスト表 → [movement.md](movement.md)）。
- 表計算向き＝**ユニット性能・エイリアス・移動タイプ**の3表（1行=1レコードのフラット表）。ステージ(json) は手書きのまま。

### 4. ステージからの参照

- **ステージは種別を名前参照**: `{ "type": "cleric", "team": 0, "col": 3, "row": 3 }`（`level`/`troops` 等は任意で上書き）。
- `StageLoader` が `UnitCatalog` 経由で解決し、`Unit.type_id` に保持（描画でスキンを引く／将来の占領＝寝返りに使う）。

### 5. 現状と将来

- 現状の実装ロスター/スキンは10種（cleric/priest/monk/bishop/paladin/archer/thief/birdman/fighter ＋ 敵専用 dragon）。残りは順次。
- `attack_range`(>1=間接)・`move_type`・`atk_air`(対地/対空) は実装済み。攻撃側は相手が飛行なら `atk_air`、地上なら `atk_ground` を使い、`atk_air=0` の駒は飛行を攻撃・反撃できない（→ [combat.md](combat.md)）。防御は単一値（`defense`）。
- 【将来】移動タイプ＝地形移動コスト・地形適性（例: 森を低コストで抜ける）はマップの地形テーブルとセットで別途設計。
- 【将来】アーキ本筋の「原本＝スプレッドシート→CSV→.tres」量産パイプライン（[../tech/architecture.md](../tech/architecture.md)）。当面は JSON で回す。
