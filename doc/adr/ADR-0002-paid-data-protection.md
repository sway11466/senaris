# ADR-0002: 有料データの保護方針（デジタル署名＋pck 暗号化）

- **ステータス**: 承認済み (Accepted)
- **決定日**: 2026-06-27
- **対象**: 有料コンテンツ（DLC ステージ／キャラ）・体験版／製品版のデータ保護

---

## コンテキスト

将来、**ステージやキャラを有料データとして販売**する可能性がある（体験版／製品版の出し分け、DLC）。
本作はデータ駆動設計で、ステージ＝`data/stages/*.tres`、キャラ＝`data/units/*.tres` の **データファイル**として表現される。
そのため次のリスクがある:

- **海賊版**: 有料データが抜き出され、買わずに出回る。→ 売上に直撃。
- **偽造・改造データ**: 改造 `.tres` や非公式データが「公式」と偽って配布され、改造キャラが蔓延したり評判を損なう。

本作は**シングルプレイ／オフライン**であり、ローカルデータの改造を**100% 防ぐことは原理的に不可能**（相手の PC 上で動くため）。
したがって目標は「完全防御」ではなく **「カジュアルな改造・流用を全滅させ、突破コストを実行ファイル改造レベルまで引き上げる」** こととする。

---

## 決定

有料データを次の2層で保護する。**両者は目的が異なるため併用する。**

1. **デジタル署名（公開鍵暗号）— 正規性の検証**
   - 出荷データに、**開発者だけが持つ秘密鍵**で署名を付す。
   - ゲーム本体に**公開鍵**を埋め込み、**署名が正しいデータのみ読み込む**。
   - 改造 `.tres` や偽造データは署名不一致で**ロード時に拒否**される。
   - 突破には実行ファイル自体の改造が必要になり、データ単体をいじる攻撃は無効化される。
   - 秘密鍵は **絶対に配布・コミットしない**（リポジトリ外で管理）。

2. **pck 暗号化（共通鍵）— 中身の秘匿**
   - エクスポート時に `.pck` を暗号鍵で暗号化し、**データマイニング／中身の流用を阻む**。
   - カジュアルな抜き出し・閲覧を防ぐ（鍵は本体内のため、本気の攻撃者には突破され得る＝ハードル上げ）。

3. **所有権チェック（エンタイトルメント）— 購入者だけに使わせる**
   - ストアに「**今これを起動してる人**がこの DLC を持っているか」を**ユーザー単位**で問い合わせる（例：`Steam.is_dlc_installed(DLC_APP_ID)`）。
   - **署名・暗号化では防げない「仲間シェア／丸ごと流出」を止める唯一の手**。
     署名は「本物か／改ざんされてないか」しか判定せず、本物ファイルが丸ごとコピーされると当然パスする。「誰が買ったか」は所有権チェックの担当（軸が違う）。
   - ストア依存の機能のため `infrastructure/platform/` に隔離する（後述）。任意の強化として**ユーザー単位暗号化**（購入者鍵でバインド）も検討余地。

### 実装上の前提（土台のみ先に用意）

- **データ読み込みを 1 か所のローダー層に集約する**。署名検証はそこに 1 枚差せば全データに効く。バラバラに `load()` を撒かない。
- **有料／無料データの境界を最初から引く**。「買った人にだけデータを配る」方式（別 depot／追加 pck）へ後から切り替えられるようにする。
- **所有権チェックを `infrastructure/platform/` に隔離**し、本体は `owns(content_id) -> bool` だけを見る（チャネル非依存）。Steam 固有のものを `domain`/`application`/`data` に焼き込まない。将来のチャネル追加（アプリストア等）は実装差し替えで済むようにする。
- **論理コンテンツ ID ⇄ ストア別 product ID のマッピング層**を用意し、`data/` には論理 ID だけ持たせる。
- **課金ラインを薄い本体に寄せすぎない**。「薄い本体＋分厚い DLC」は守りが所有権チェック 1 枚に集中し、最も脆くなるため。

### スコープ外（やらないこと）

- 完全な耐タンパー（実行ファイル改造まで防ぐ DRM 要塞）は**狙わない**。費用対効果が崩壊するため。
- オンライン／ランキング等を将来導入する場合は、その時に**サーバ側検証**を別途設計する（本 ADR の対象外）。

---

## 根拠と影響 (Consequences)

### 利点

- **改造・偽造データを構造的に弾ける**（署名）。データ単体の攻撃が成立しなくなる。
- **中身の流用・データマイニングを抑止**（暗号化）。
- **仲間シェア／丸ごと流出を購入者単位で弾ける**（所有権チェック）。
- 「読まれない（暗号化）＋偽れない（署名）＋買った人だけ（所有権）」の三点で、カジュアル層の不正をほぼ排除できる。
- ローダー集約・所有権チェックの隔離により、保護機構の追加・変更／チャネル追加が一点で済む。

### トレードオフ・留意点

- **本気の攻撃者（実行ファイル改造）は最終的に突破し得る。** これは許容する（目標はカジュアル阻止）。
- 秘密鍵の管理責任が発生する（紛失＝署名し直し不可、漏洩＝偽造可能）。
- 署名検証・復号のぶんロード処理が増える（ターン制 SLG では実用上問題ない想定）。

### 実装タイミング

- **着手は「有料データが実在し始めてから」で間に合う。** 現時点ではステージ・キャラともデータ化前。
- ただし **ローダー集約と有料/無料境界**は、データ化を始める段階で**最初から織り込む**（後付けは高コスト）。

## 検討した代替案

| 案 | 不採用／補助の理由 |
| --- | --- |
| **何もしない（生データ出荷）** | メモ帳で改造可能。有料化するなら最低限の防御が要る。 |
| **暗号化のみ** | 中身は隠せるが、改造・偽造の「正規判定」ができない。署名と併用する。 |
| **署名のみ** | 偽造は防げるが中身は読める（データマイニング可）。暗号化と併用する。 |
| **完全 DRM / 耐タンパー要塞** | 個人開発には過剰。突破は時間の問題で、投資効率が悪い。 |
| **サーバ側検証** | シングルプレイでは検証相手が居らず不要。オンライン導入時に再検討。 |

- 用語整理:
  - **署名（公開鍵暗号）** = 「正規かどうか」を判定。秘密鍵で署名し、公開鍵で検証。
  - **暗号化（共通鍵）** = 「読めるかどうか」を制御。中身の秘匿。
