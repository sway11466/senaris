# i18n（多言語対応）

テキストの多言語化の方針。Steam での海外販売を前提に、プレイヤーが読む文字列はすべて翻訳キー経由にする。

## 方針

- 正本は日本語。`fallback=ja` で、未翻訳キーは日本語にフォールバックする。
- リリース時の対応言語は ja＋en の2言語。以降の言語追加は翻訳CSVに列を足すだけで済む構造を維持する（中国語圏等はフォント整備が別途要るため、実施はリリース後に判断）。
- プレイヤーが読む文字列を GDScript・データCSV・ステージJSON に直書きしない。すべて翻訳キーにして `tr()` で解決する。開発用のログ・push_error 文言は対象外（日本語直書きでよい）。

## 仕組み（実装済みの土台）

- 翻訳の正本は `data/i18n/*.csv`（1行目 `keys, ja, en, …`）。Godot 標準の翻訳インポータが `.translation`（言語別・生成物だが git 管理）を生成し、`project.godot` の `locale/translations` に登録する。
- コードは `tr("キー")`、ステージJSON・マニフェストはテキスト欄に翻訳キーを書く（例: `"title": "t1.title"`）。
- 話者名・会話の実装詳細 → [../campaign/authoring.md](../campaign/authoring.md)。

## CSV の使い分け（罠）

- 翻訳CSV（`data/i18n/`）: 翻訳インポータを使う（既定のまま）。
- データCSV（`data/units/` 等の性能表）: `importer="keep"` 必須。既定のままだと翻訳インポータが食って壊れる。
- 逆向きの使い分けなので、CSV を新設するときはどちらの系統かを最初に決める。

## キー命名規約

- 基本形は `<系統>.<id>.<項目>`。系統ごとに CSV ファイルを分ける。
  - `dialogue.csv` — 会話・話者名（`t1.st3.intro.1` / `char.cap.name`）
  - `campaigns.csv` — 冒険譚・ステージのタイトル/説明（`t1.title`）
  - `units.csv` — ユニット・地形・移動タイプの表示名（`unit.cleric.name` / `terrain.road.name` / `movement.ground.name`）※未作成・下記移行
  - `ui.csv` — UI 文言（`ui.hud.end_turn` / `ui.info.troops`）※未作成・下記移行
- キーは追加のみとし、意味変更での使い回しをしない（翻訳済み文字列が古い意味のまま残る事故を防ぐ）。

## 表示名の扱い（データCSVの name 列）

- ユニット・地形などの表示名は、id からの規約キー（`unit.{skin_id}.name` 等）で翻訳CSVから引く。画像の autowire（`assets/units/{skin_id}/…`）と同じ「規約で解決する」方式。
- データCSVの `name` 列は表示に使わず、開発用メモに降格する（列自体は残してよい。2行目の日本語ラベル行と同じ扱い）。
- 未実施＝現状は `UnitSkin.name`／`TerrainSkin` の表示名がデータCSVの日本語をそのまま表示している。移行 → [backlog feature-12](../backlog.md)。

## UI 文言

- 新規の UI はプレイヤー向け文字列を `tr()` 必須とする。
- 既存の直書き（HUD のボタン・情報パネルのラベル・勝敗表示など）は一括でキー化する。移行 → [backlog feature-12](../backlog.md)。

## テスト

- `tests/unit/test_i18n.gd`・`test_i18n_translation.gd` が翻訳CSVの整合（キーの実在・ja/en 両方が埋まっていること）を守る。翻訳キーを追加したらテストが検出範囲に含むことを確認する。
- テスト方針の全体 → [testing.md](testing.md)。
