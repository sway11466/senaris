# アーキテクチャ設計メモ

対象：ヘックス制タクティクスSLG（Godot 4 / Steam）
役割：レイヤー構成・モジュール構成・依存ルールの決定事項を記録する生きたメモ

---

## 大方針 【確定】

ルールエンジン（Domain）をGodotのノードから分離する。

戦闘解決が「経験→包囲→支援→地形」の決定的パイプラインであり、包囲・陣形スキルもヘックス上の空間クエリであるため、ここを純ロジックとして独立させる。これにより：

- 中断セーブが「状態を丸ごと直列化するだけ」で済む
- 戦闘計算をエンジン起動なしでテストできる（補正値の検証が速い）
- 将来のリプレイ・AI先読みに同じロジックを再利用できる

---

## レイヤー構成 【確定】

依存は上から下への一方向（`presentation → application → domain → data`）。

```
┌─────────────────────────────────────┐
│ Presentation（Godotノード/シーン）    │ 入力・描画・アニメ・UI・SE
├─────────────────────────────────────┤
│ Application（ゲーム進行）             │ ターン管理・フェーズ・コマンド処理
├─────────────────────────────────────┤
│ Domain（純ロジック / Godot非依存）    │ 戦闘解決・包囲・陣形スキル・占領
├─────────────────────────────────────┤
│ Data（リソース定義）                 │ ユニット表・陣形スキルレシピ・地形・テーマ
└─────────────────────────────────────┘
```

## 依存ルール 【確定】

### 黄金ルール（1つだけ覚える）

依存は `presentation → application → domain → data` の一方向。逆向き禁止。
`domain` と `data` はGodotのノードを一切知らない。

### それを守る具体規約

1. `domain/` と `data/` の型定義（`*_type.gd` 等）は `extends Node` 禁止。`extends Resource` か `extends RefCounted` のみ。← 最も効くチェック
   - 型定義は専用の `schema/` に集約せず、各機能フォルダにデータと同居させる（型とデータはセット）。【更新: 旧 data/schema/ を廃止】
2. `domain/` から `$`・`get_node`・`Input.`・シーン読み込みを呼ばない
3. Presentationは状態を直接書き換えない。下りは「コマンド」をApplicationに投げ、上りはApplicationの「シグナル」を受けて描くだけ
4. セーブ対象は `battle_state.gd` に集約。見た目の状態はセーブに含めず、状態から再構築する

### Presentation ↔ Application の会話

- 下り：`presentation` が `MoveCommand{unit, from, to}` のようなコマンドを投げる
- 上り：`application` が `unit_moved` / `combat_resolved` / `formation_triggered` などのシグナルを純データで発行 → `presentation` が受けてアニメ

状態の真実は `domain/battle_state.gd` ひとつ。テスト・中断セーブ・将来のリプレイが全てこの一点に乗る。

## 各レイヤーの決定事項

### 言語 【確定】
- GDScript（型付き）を採用。テストフレームは GUT（方針・実行方法 → [testing.md](testing.md)）
- Domain層からノードを触らない規約で層を守る（コンパイラの代わりに規約で担保）

### データ形式 【確定】
- 原本＝スプレッドシートで管理 → CSV書き出し → 取り込みで型付きResourceに変換
- 入れ子データ（陣形スキルレシピ等）＝ JSON か Resource
- 実行時は常に型付きオブジェクト（custom Resource）を扱う
- テーマ差し替え＝ステータスは原型に1回だけ、名前・スプライトのみ上書きレイヤー（複製しない）

#### CSV→データ生成のバリデーション（基本方針）

CSV正本からデータを生成するツールは複数ある（`data/*/convert.gd`：terrain / units / ai / movement …＋共有 `csv_util`）。壊れた生成物がそのままゲームに載らないよう、十分なバリデーションを各変換ツール＋共有ユーティリティの必須責務とする。

- 検出対象: 列の過不足／必須列の空セル／ドメイン外の値（未知ラベル・型不一致・範囲外）。
- 欠損・不正はデータのバグとして扱い、開発時に落とす（`push_error`／`assert`／GUT 失敗）。生成時（convert 実行）・読込時（`*_catalog`）・テストのいずれかで必ず検出し、プレイヤーに届く経路では出さない。export 版で消える `assert` は runtime 保険として低コスト。
- 現状の不足: `csv_util.read_table` は列不足の行を黙ってスキップし、スキーマ検証を持たない。ここの検出化＋各 convert への必須列チェック追加が整備タスク。最初の適用例は AIプリセットの全軸そろい検証（doc/gdd/ai.md「既定と省略のポリシー」）。

### セーブ 【確定】
- 進捗セーブ（マップ単位）＋中断セーブ（Domain状態を丸ごと直列化）の二本立て
- ターン毎オートセーブは中断セーブの応用として後から派生可能

---

## 未決・保留

- 【未決】戦闘の決定性（乱数の有無）。中断セーブの正確さと将来のリプレイに影響。乱数があるならシード/状態もセーブ対象になる
