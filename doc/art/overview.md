# 画像準備メモ（2026-06-08）

ファンタジー・ヘックス戦術SLGの、ユニット等のグラフィック準備に関する検討メモ。

> 色・形のルールと制作スペックは [visual-identity.md](visual-identity.md)（生成前に決める正本）。陣営ごとの個体特徴は `assets/units/source/{group}/style.md`（味方27種＝[player/style.md](../../assets/units/source/player/style.md)）。

---

## 1. 検討の目的

- システムの面白さは Nectaris で確信済み。今回検証したいのはファンタジーの味付けが刺さるか。
- 評価者は自分。他人に見せる前提ではない。
  - → 完成度より方向性の手応えが大事。荒くても、自分が見て世界が立ち上がればOK。

## 2. アプローチ：縦スライス

- 横（全23ユニット × テーマ別リスキン）に広げる前に、縦に薄く切って代表だけ本番想定クオリティで作り、味を見る。
  - 理由: 味付けの良し悪しを決めるのはトーンとユニットの読み替えであり、全ユニットの作り込みではない。刺さると確信してから横展開すれば手戻りが最小。
- 最小構成は味方1体 ＋ 敵1体（例: クレリック＋ドラゴン）。
  - 自分が評価者なので、1組見た時点で「もう分かった」となる可能性が高い。地形の一角もあれば雰囲気判定はより確実。

## 3. 盤面でのユニット表現（候補）

| 方式 | 強み | 弱み |
| --- | --- | --- |
| デフォルメ2Dキャラ（チビキャラ） | 世界観との相性◎。味方=人/敵=モンスターが一目で伝わる | 制作コスト最大。ヘックスが小さいと細部が潰れやすい |
| 武器・シンボルアイコン | 制作が軽い・識別性が高い。パズル把握に向く | 没入感がキャラ絵に劣る。敵味方差は色/枠で工夫が要る |
| 折衷（盤面は軽く＋詳細パネルで立ち絵） | 識別性と魅力を両取り。立ち絵は1枚で済む | — |

- どの方式にするかは未決。生成して見比べてから決めるのもあり。

## 4. 制作手段

- 生成AI（Gemini / Nano Banana）を採用。

### Geminiの能力（調査結果）
- チビキャラ・アニメ調イラストは生成可能。髪型・目・小道具など具体指定で個性を出せる。
- 画像モデルは Nano Banana（Gemini 2.5 Flash Image）。最新は Nano Banana 2（2026-02-26 リリース）。上位に Nano Banana Pro。
- 最大の利点はキャラの一貫性維持。同じキャラの特徴（髪型・服装・表情）を保ったまま別シーン・別ポーズを生成できる。
  - → 統一画風で多数ユニットを揃える今回の要件に合致。

### 配布前提の注意点
- プロンプトに既存キャラ・商標・特定アーティスト名を入れない。著作権侵害の責任は原則ユーザー側が負う。Nectaris由来の固有名詞や既存ゲームキャラを連想させる指示は避ける。
- 本格的な商用利用時は、著作権補償のある環境（Vertex AI 等）を検討。趣味の味見段階は無料版で可。
- 生成画像には AI生成を示す SynthID が自動付与される。これとは別に Steam の AI開示ルールがあり、配布段階で要確認。

## 4.5 画像データ構造（実装）

性能（`UnitType`）と分離した `UnitSkin` が名前・説明・画像を持つ（→ [../gdd/units.md](../gdd/units.md)）。画像はスロット制：

| スロット | 用途 | 未用意時のプレースホルダ |
|---|---|---|
| `map` | 盤上のユニット表示 | ベタ塗り＋名前の先頭2文字（例: クレリック→「クレ」） |
| `combat` | 戦闘演出の表示 | ベタ塗り＋フルネーム |

- `images = { "map": "res://...", "combat": "res://..." }`。未設定はプレースホルダ（盤は名前を描く）。アートが来たらパスを入れるだけで描画が画像へ切り替わる（コード不変）。
- 行動前/行動後は当面 `map` 1枚＋盤側のグレー化（行動済みは暗くする処理が既にある）。専用画像が要るなら `map_active`/`map_done` にスロットを割ればよい。
- `combat` の描画（戦闘演出シーン）はまだ無し＝スロットだけ先に用意。
- 将来スロット候補: `portrait`（会話/図鑑の立ち絵）、`icon`。

## 5. 次のアクション

- 味方1体・敵1体を生成するためのプロンプト設計から着手 → 実際に味の評価へ進む。
- 盤面表現方式は、生成サンプルを見比べてから決定でも可。
